# 多语言文档翻译技能

本技能提供项目多语言文档的专业翻译能力，自动检测项目配置、同步目录结构、执行高质量翻译并验证完整性。

## 触发条件

当检测到以下情况时，主动激活此技能：

- 用户提及翻译文档："翻译文档"、"多语言文档"、"国际化文档"、"i18n"、"文档本地化"
- 用户提及具体翻译需求："把文档翻译成英语"、"生成日语版本文档"
- 用户提及翻译相关操作："更新翻译"、"同步多语言文档"
- 英文触发词："translate documentation"、"i18n docs"、"localize docs"

---

## 第一阶段：项目配置检测

在执行翻译前，需全面检测项目的文档配置和结构。

### 1.1 检测静态站点生成器

检查项目使用的文档工具，识别配置文件：

| 工具 | 配置文件 | 默认语言配置位置 |
|------|----------|------------------|
| VitePress | `.vitepress/config.mts` 或 `config.ts` | `locales.root` 或 `defaultLang` |
| Hugo | `hugo.toml` / `config.toml` | `defaultContentLanguage` |
| Jekyll | `_config.yml` | `lang` 或 `defaults.lang` |
| Hexo | `_config.yml` | `language` |
| Docusaurus | `docusaurus.config.js` | `i18n.defaultLocale` |
| Next.js Nextra | `next.config.js` 或 theme config | `i18n.defaultLocale` |

**检测流程**：

1. 检查项目根目录是否存在上述配置文件
2. 解析配置文件，提取默认语言设置
3. 识别文档目录结构模式

### 1.2 识别文档目录结构模式

根据项目配置和现有文档结构，识别多语言目录模式：

**模式一：子目录模式（推荐）**

```
docs/
├── zh/           # 中文文档
│   ├── guide/
│   │   └── getting-started.md
│   └── index.md
├── en/           # 英文文档
│   ├── guide/
│   │   └── getting-started.md
│   └── index.md
└── .vitepress/
    └── config.mts
```

**模式二：根目录模式**

```
docs/
├── guide/
│   └── getting-started.md    # 默认语言
└── index.md
en/
├── guide/
│   └── getting-started.md    # 英文翻译
└── index.md
```

**模式三：文件名后缀模式**

```
docs/
├── guide/
│   ├── getting-started.md      # 默认语言
│   ├── getting-started.en.md   # 英文翻译
│   └── getting-started.ja.md   # 日语翻译
└── index.md
```

### 1.3 确定默认语言

按以下优先级确定文档的默认语言：

1. 从配置文件中读取默认语言设置
2. 检查现有文档内容，通过语言特征识别
3. 如果无法确定，询问用户确认

### 1.4 确定目标语言

按以下流程确定目标语言：

1. 用户在请求中明确指定目标语言 → 使用指定语言
2. 从项目配置中读取支持的语言列表 → 确认翻译目标
3. 无法确定时 → 使用 AskUserQuestion 工具询问用户

## 第二阶段：目录结构同步

确保目标语言目录结构与源语言完全一致。

### 2.1 扫描源文档结构

递归扫描默认语言目录，记录所有文档文件：

```markdown
## 源文档结构

- 默认语言：zh
- 文档目录：docs/zh
- 文件总数：15

### 文件列表
1. docs/zh/index.md
2. docs/zh/guide/getting-started.md
3. docs/zh/guide/configuration.md
4. docs/zh/api/overview.md
...
```

### 2.2 创建目标目录结构

在目标语言目录中创建与源语言一致的目录结构：

**执行步骤**：

1. 获取源语言目录下所有文件的相对路径
2. 在目标语言目录下创建对应的目录结构
3. 确保目录层级完全一致

**示例**：

```
源语言：zh，目标语言：en

源结构：                    目标结构：
docs/zh/                    docs/en/
├── guide/                  ├── guide/
│   └── intro.md    →       │   └── intro.md (待翻译)
└── api/                    └── api/
    └── reference.md            └── reference.md (待翻译)
```

### 2.3 验证目录结构

完成翻译后，验证目录结构完整性：

- 检查所有源文件是否都有对应的目标语言文件
- 检查目录层级是否一致
- 报告缺失的文件或目录

## 第三阶段：翻译执行

执行高质量的文档翻译，确保"信、雅、达"的翻译标准。

### 3.1 翻译原则

**信（忠实）**

- 准确传达原文含义，不增减信息
- 保持技术术语的准确性
- 保留原文的代码示例、链接、图片等元素

**雅（优雅）**

- 使用目标语言的自然表达方式
- 遵循目标语言的写作习惯
- 保持文档的专业性和可读性

**达（通顺）**

- 确保译文流畅易懂
- 避免翻译腔和生硬表达
- 适当调整句式以符合目标语言习惯

### 3.2 翻译规则

**技术术语处理**

| 情况 | 处理方式 | 示例 |
|------|----------|------|
| 通用技术术语 | 保留英文或使用标准译名 | API → API，算法 → Algorithm |
| 项目特定术语 | 保持一致性 | 组件名、配置项名称不翻译 |
| 代码相关 | 完全保留 | 变量名、函数名、命令不翻译 |
| 链接路径 | 根据情况调整 | 内部链接可能需要调整路径 |

**Markdown 元素处理**

```markdown
# 标题翻译
## Heading Translation

正文内容翻译，保持 **加粗** 和 `代码` 格式。

- 列表项翻译
- 保持列表结构

> 引用块翻译

[链接文本翻译](url-保持不变)

![图片替代文本翻译](image-path.png)

| 表头翻译 | 表头翻译 |
|----------|----------|
| 内容翻译 | 内容翻译 |

```language
// 代码块不翻译，但注释可翻译
const example = "code"; // 示例代码
```

**Frontmatter 处理**

```yaml
---
title: 翻译标题
description: 翻译描述
lang: en  # 更新语言标识
---
```

### 3.3 增量翻译策略

基于文件修改时间进行增量翻译：

**检测流程**：

1. 获取源文件的最后修改时间 `source_mtime`
2. 获取目标文件的最后修改时间 `target_mtime`
3. 比较：
   - `source_mtime > target_mtime` → 需要更新翻译
   - 目标文件不存在 → 需要新建翻译
   - `source_mtime <= target_mtime` → 跳过（翻译已是最新）

**执行策略**：

```markdown
## 翻译计划

### 需要新建翻译 (3个文件)
- docs/en/guide/new-feature.md
- docs/en/api/new-endpoint.md
- docs/en/changelog.md

### 需要更新翻译 (2个文件)
- docs/en/index.md (源文件已更新)
- docs/en/guide/configuration.md (源文件已更新)

### 跳过翻译 (10个文件)
- docs/en/guide/getting-started.md (已是最新)
- ...
```

### 3.4 翻译执行流程

**单文件翻译流程**：

1. 读取源文件内容
2. 解析 Markdown 结构
3. 逐段翻译内容（保留代码块、链接等）
4. 更新 frontmatter 中的语言标识
5. 写入目标文件
6. 记录翻译日志

**批量翻译流程**：

1. 扫描并分类文件（新建/更新/跳过）
2. 展示翻译计划，确认用户意图
3. 按顺序执行翻译
4. 实时报告进度
5. 完成后生成翻译报告

## 第四阶段：翻译验证

完成翻译后，进行全面的质量验证。

### 4.1 结构完整性验证

检查目标语言目录结构是否完整：

**验证清单**：

- [ ] 所有源文件都有对应的目标语言文件
- [ ] 目录层级完全一致
- [ ] 文件命名规范一致
- [ ] 没有遗漏的文件

**验证报告示例**：

```markdown
## 结构验证报告

### 通过项
✅ 目录结构一致
✅ 文件数量匹配 (15/15)
✅ 文件命名规范正确

### 缺失文件
❌ docs/en/advanced/performance.md (源文件存在，目标文件缺失)

### 多余文件
⚠️ docs/en/deprecated/old-feature.md (源文件已删除，目标文件仍存在)
```

### 4.2 内容完整性验证

检查翻译内容的完整性：

**验证项目**：

- [ ] 标题数量一致
- [ ] 段落数量一致
- [ ] 代码块数量一致
- [ ] 链接数量一致
- [ ] 图片数量一致
- [ ] 表格结构一致

### 4.3 翻译质量检查

**基础检查**：

- 是否有未翻译的段落
- 是否有翻译错误或歧义
- 技术术语是否一致
- 格式是否正确

**术语一致性检查**：

建立术语表，确保同一术语在所有文档中的翻译一致：

| 源术语 | 目标语言翻译 | 备注 |
|--------|--------------|------|
| 组件 | Component | 统一使用 |
| 配置 | Configuration | 统一使用 |
| 钩子 | Hook | 统一使用 |

### 4.4 链接有效性验证

检查翻译文档中的链接：

- 内部链接：确保目标语言版本的对应文件存在
- 外部链接：确保链接仍然有效
- 锚点链接：确保目标标题存在

## 常见场景处理

### 场景一：首次翻译整个项目

```
用户：把这个项目的文档翻译成英语

Agent：
1. 检测项目配置：发现使用 VitePress，默认语言为中文
2. 扫描文档目录：发现 20 个文档文件
3. 创建目标目录：docs/en/
4. 确认翻译计划：翻译全部 20 个文件
5. 执行翻译
6. 验证完整性
7. 生成报告
```

### 场景二：增量更新翻译

```
用户：更新英文文档

Agent：
1. 检测项目配置
2. 对比修改时间
3. 发现 3 个文件需要更新
4. 展示更新计划
5. 执行增量翻译
6. 验证并报告
```

### 场景三：翻译单个文件

```
用户：把 docs/zh/guide/getting-started.md 翻译成日语

Agent：
1. 读取源文件
2. 检查目标目录是否存在
3. 执行翻译
4. 写入 docs/ja/guide/getting-started.md
5. 验证翻译质量
```

### 场景四：添加新语言支持

```
用户：添加日语文档支持

Agent：
1. 检测现有文档结构
2. 创建 docs/ja/ 目录
3. 复制目录结构
4. 翻译所有文档
5. 更新项目配置文件（如 VitePress config）
6. 验证并报告
```

## 配置文件更新

翻译完成后，根据需要更新项目配置文件。

### VitePress 配置更新

```typescript
// .vitepress/config.ts
export default defineConfig({
  locales: {
    root: {
      label: '简体中文',
      lang: 'zh',
    },
    en: {
      label: 'English',
      lang: 'en-US',
    },
    ja: {
      label: '日本語',
      lang: 'ja-JP',
    }
  }
})
```

### Hugo 配置更新

```toml
# hugo.toml
defaultContentLanguage = 'zh'
languages:
  zh:
    title: 项目文档
    weight: 1
  en:
    title: Project Documentation
    weight: 2
```

## 注意事项

1. **保持一致性**：同一术语在所有文档中的翻译必须一致
2. **尊重原文**：准确传达原文含义，不随意增删内容
3. **本地化表达**：使用目标语言的自然表达方式
4. **技术准确性**：技术术语、代码示例保持准确
5. **格式规范**：保持 Markdown 格式的正确性
6. **链接维护**：确保所有链接在翻译后仍然有效
